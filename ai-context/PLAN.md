## Findings (security workstream)

A focused review of the action-execution surface and provider HTTP layer
turned up four issues, captured in detail in
ai-context/security/PATCH_PLAN.md. Headlines:

- SEC-001 (P1 High) — Command-injection bypass in src/actions/execute.ts.
  `execa(action.command, { shell: true, ... })` combined with a
  prefix-anchored allow list and an incomplete deny list lets an LLM-produced
  command satisfy `^pnpm test\b` and chain arbitrary code via shell
  metacharacters (`;`, `&&`, `>`, `$()`, backticks). The "sandbox" is just
  `os.tmpdir()`, so the attacker process inherits the user's environment.
- SEC-002 (P2 Med) — Gemini provider does not redact API responses before
  storing them in `rawOutput` (other providers do).
- SEC-003 (P2 Med) — Gemini API key sent as `?key=` query parameter rather
  than `x-goog-api-key` header.
- SEC-004 (P3 Low) — Documentation gap on sandbox isolation guarantees.

No P0 build/test breakage was detected. No critical bugs were filed in
ai-context/bug. The next safest, highest-impact action is SEC-001.
